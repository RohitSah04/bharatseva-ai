## 1. Structured JSON Logging

Every log line emitted by the application is a **single-line JSON object**. No multi-line stack traces, no free-form strings. Human-readable formatting is enabled only in the `development` environment via a `LOG_FORMAT=pretty` env var.

### Standard Log Fields (present on every line)

```json
{
  "timestamp":   "2025-07-11T10:23:45.123Z",   // ISO-8601 UTC
  "level":       "INFO",                         // DEBUG | INFO | WARNING | ERROR | CRITICAL
  "service":     "bharatseva-api",               // fixed string; change for worker processes
  "environment": "production",                   // from APP_ENV env var
  "request_id":  "a3f8c2d1-...",                 // UUID v4, see §2
  "message":     "...",                          // human-readable event description
  "module":      "routes.eligibility",           // Python module path
  "line":        42                              // source line number
}
```

### Request-Scoped Additional Fields

Added by the request-logging middleware on the **incoming request line**:

```json
{
  "event":       "request_in",
  "method":      "POST",
  "path":        "/api/v1/eligibility/check",
  "user_id":     "uuid-of-caller",               // null for unauthenticated
  "role":        "citizen",
  "ip":          "redacted",                     // IP logged for rate-limit debugging only
  "user_agent":  "Mozilla/5.0 ..."
}
```

Added by the request-logging middleware on the **outgoing response line**:

```json
{
  "event":         "request_out",
  "method":        "POST",
  "path":          "/api/v1/eligibility/check",
  "status_code":   200,
  "latency_ms":    142,
  "degraded":      false
}
```

### Agent-Invocation Log Fields

Written by the Orchestrator for every agent call (in addition to the DB row in `agent_logs`):

```json
{
  "event":          "agent_call",
  "agent_name":     "eligibility",
  "latency_ms":     890,
  "confidence":     0.87,
  "fallback_used":  false,
  "escalated_to":   null
}
```

### Error Log Fields

```json
{
  "level":       "ERROR",
  "event":       "unhandled_exception",
  "exception":   "ValueError",
  "message":     "eligibility_rules_json is malformed for scheme abc-123",
  "traceback":   "..."    // only in development; omitted in production to avoid leaking internals
}
```

## 2. Request-ID Strategy

Every HTTP request that enters the Flask application is assigned a **unique `X-Request-ID`** header value:

1. If the incoming request already has an `X-Request-ID` header (set by a load balancer or CDN), it is reused as-is.
2. If no such header is present, a new UUID v4 is generated by the middleware.
3. The request ID is stored in Flask's `g` object for the duration of the request so that any function deeper in the stack can include it in log lines without explicitly threading it through function parameters.
4. The `X-Request-ID` is included in **every response header** so that a client (or support engineer) can correlate a specific request to its log lines.
5. The request ID is propagated to every agent invocation and stored in `agent_logs.request_id`. This makes it possible to retrieve every DB row and log line for a single citizen interaction from one query.

**Correlation query example (SQLite):**
```sql
SELECT * FROM agent_logs WHERE request_id = 'a3f8c2d1-...';
SELECT * FROM eligibility_checks WHERE agent_log_id IN (
    SELECT id FROM agent_logs WHERE request_id = 'a3f8c2d1-...'
);
```

## 3. Health Check Endpoints

### GET /api/v1/health (Liveness)

**Semantics:** Is the process alive and able to receive requests?
**Returns 200** as long as the Flask application is running. Does **not** check DB connectivity or AI provider reachability.
**Use case:** Load balancer / container orchestrator liveness probe. If this returns non-200, restart the instance.

```json
{
  "status": "ok",
  "timestamp": "2025-07-11T10:23:45Z"
}
```

---

### GET /api/v1/health/ready (Readiness)

**Semantics:** Can this instance serve real traffic — are all dependencies reachable?
**Checks performed:**
1. SQLite (or PostgreSQL): execute `SELECT 1` and verify response within 1 second.
2. AI provider (watsonx.ai): ping the `/health` endpoint of the watsonx.ai API or a lightweight models-list call; timeout = 3 seconds.
3. Vector store (ChromaDB): attempt a trivial query; timeout = 2 seconds.

**Returns 200** if all checks pass. **Returns 503** if the DB check fails (cannot serve any traffic). Returns 200 with individual component marked `"degraded"` if AI provider or vector store is unreachable (application can still serve cached/rule-based responses).

```json
{
  "status": "ready",
  "checks": {
    "db":           { "status": "ok",       "latency_ms": 3 },
    "ai_provider":  { "status": "degraded", "latency_ms": 3001, "detail": "timeout" },
    "vector_store": { "status": "ok",       "latency_ms": 12 }
  },
  "timestamp": "2025-07-11T10:23:45Z"
}
```

503 example (DB unreachable):
```json
{
  "status": "not_ready",
  "checks": {
    "db":           { "status": "error", "detail": "connection refused" },
    "ai_provider":  { "status": "ok",    "latency_ms": 240 },
    "vector_store": { "status": "ok",    "latency_ms": 8 }
  },
  "timestamp": "2025-07-11T10:23:45Z"
}
```

## 4. What Gets Audited

The following events are considered **security-significant or AI-significant** and are written to permanent, append-only audit stores. These rows must **never be updated or deleted** — audit integrity is non-negotiable.

### 4.1 Eligibility Checks → `eligibility_checks` table

Every time the Eligibility agent produces a verdict, the following is persisted:

| Field | What it captures |
|---|---|
| `user_id` | Which citizen received the verdict |
| `scheme_id` | Which scheme was evaluated |
| `verdict` | ELIGIBLE / NOT_ELIGIBLE / PARTIALLY_ELIGIBLE |
| `confidence_score` | AI confidence at decision time |
| `reasoning` | Plain-language justification (translated to citizen's language) |
| `sources_json` | Exact KB chunks used, with kb_version |
| `fallback_used` | Whether the verdict came from AI or rule-based fallback |
| `checked_at` | Timestamp — immutable |

**Why:** If a citizen is told they are not eligible and contests the decision, this row is the evidence. It must reflect exactly what the AI returned, including the source versions used.

### 4.2 Agent Invocations → `agent_logs` table

Every agent call by the Orchestrator:

| Field | What it captures |
|---|---|
| `agent_name` | Which agent was called |
| `request_id` | The HTTP request that triggered this call |
| `input_json` | Sanitised input (PII redacted) |
| `output_json` | Full AgentResponse JSON |
| `confidence` | Agent's stated confidence |
| `latency_ms` | Execution time |
| `fallback_used` | Whether the AI call failed and was replaced by rule-based logic |
| `escalated_to` | If the agent passed responsibility to another agent |
| `created_at` | Timestamp — immutable |

**Why:** This is the source of truth for the Admin Portal's agent-performance panel. It also supports investigation of any AI output a citizen disputes.

### 4.3 Application Status Changes → `applications.status_history_json`

Every time a status transition is recorded on an application tracker row, the history array is appended (never overwritten):

```json
[
  { "status": "NOT_STARTED", "changed_at": "2025-07-10T08:00:00Z", "note": "created on goal activation" },
  { "status": "IN_PROGRESS",  "changed_at": "2025-07-12T14:30:00Z", "note": "citizen confirmed documents ready" }
]
```

### 4.4 Goal Plans → `citizen_goals.generated_plan_json`

A Copilot plan is written once and never mutated. If a citizen re-runs the planner, a new row is created (the old one is archived with `status = ARCHIVED`). This ensures every plan the citizen received is retrievable, with the KB version it was based on.

### 4.5 Security Events → Application Logs

The following security events are logged at WARNING or ERROR level with the structured JSON format defined in §1:

| Event | Level |
|---|---|
| Failed login attempt (wrong password) | WARNING |
| Repeated failed logins (> 5 in 10 min for same email) | ERROR |
| JWT signature validation failure | WARNING |
| RBAC denial (403 issued) | WARNING |
| Rate limit hit (429 issued) | WARNING |
| File upload MIME type mismatch | WARNING |
| Prompt injection pattern detected in input | ERROR |
| Admin demo-reset executed | INFO (with admin user_id) |

## 5. Metrics and Alerting (Phase 2 Reference)

The following metrics should be exposed via a `/metrics` endpoint (Prometheus format) in Phase 2:

- `http_requests_total{method, path, status_code}` — request counter
- `http_request_duration_ms{method, path}` — histogram, p50/p95/p99
- `agent_call_total{agent_name, fallback_used}` — agent invocation counter
- `agent_call_duration_ms{agent_name}` — agent latency histogram
- `eligibility_verdict_total{verdict}` — verdict distribution
- `active_citizen_goals_total` — gauged count of ACTIVE goals
- `db_query_duration_ms{table, operation}` — DB latency histogram

In Phase 1, these can be computed from `agent_logs` and application logs in the Admin Portal analytics queries.

## 6. Log Retention Policy

| Log type | Retention | Rationale |
|---|---|---|
| Application logs (stdout/file) | 90 days rolling | Debugging, incident response |
| `agent_logs` table | Indefinite (Phase 1: no TTL) | Audit, AI accountability |
| `eligibility_checks` table | Indefinite | Citizen rights, dispute resolution |
| `chat_history` table | 12 months | Conversation continuity, analytics |
| Security event logs | 12 months | Compliance, incident investigation |

Phase 2 will introduce log archiving to object storage for long-term retention beyond what the DB can hold cheaply.
